# GoalMate 后续完整实施计划

> 文档性质：后续开发的唯一执行清单与进度台账
> 创建日期：2026-06-19
> 当前状态：实施中；P0-1、P0-2 已完成，P0-3 代码完成并等待真实 DeepSeek 预发布验证；P1-2 仓库实现完成并等待生产密钥和历史数据迁移演练
> 适用范围：`SPEC.md` 中尚未完整落地或仅完成基础版、预留版的能力

## 1. 文档目标

本文档用于指导 GoalMate 从“已完成 MVP 和完整版第一阶段基础闭环”演进到可稳定部署、可小范围真实运营、可持续商业化的生产版本。

后续开发必须以本文档为执行依据：

1. 开始工作前，先确认对应工作项的目标、依赖、范围和验收标准。
2. 开始实现时，将工作项状态从 `NOT_STARTED` 更新为 `IN_PROGRESS`，填写开始日期。
3. 实现过程中持续勾选任务，不允许只在全部完成后一次性补写。
4. 完成代码后记录迁移、接口、页面、配置、测试和文档变更。
5. 只有全部 Definition of Done 和验收命令通过，才能标记为 `DONE`。
6. 若受外部账号、密钥、资质或服务商限制，标记为 `BLOCKED`，并写明解除条件。
7. 需求发生变化时，先修改本文档，再修改代码，保证计划与实现一致。

## 2. 状态定义

| 状态 | 含义 |
| --- | --- |
| `NOT_STARTED` | 尚未开始 |
| `READY` | 依赖已满足，可以开始 |
| `IN_PROGRESS` | 正在设计、开发或验证 |
| `BLOCKED` | 因明确外部条件无法继续 |
| `PARTIAL` | 已完成可运行子集，但未满足完整验收标准 |
| `DONE` | 实现、测试、文档和验收全部完成 |
| `DEFERRED` | 经确认后推迟，不属于当前发布范围 |

## 5. P0：核心生产闭环

### P0-1 自动提醒调度

**状态：** `DONE`
**目标：** 用户无需在 Web 手动点击“生成提醒”，系统能够按用户时区、提醒时间、渠道和提醒类型自动创建并发送提醒。

#### 功能需求

- 提供应用内定时调度器，定期扫描应发送提醒的用户。
- 支持 `Asia/Shanghai`，同时保留任意 IANA 时区能力。
- 每个用户按照自己的 `reminderTime` 触发，避免所有用户集中在同一分钟处理。
- 支持提醒渠道：`EMAIL`、`WECHAT`；`WEB` 仅记录站内提醒时保留。
- 支持提醒类型：
  - 每日学习任务。
  - 当天未打卡。
  - 容错次数即将耗尽。
  - 偏差预警。
  - 救援任务创建。
  - 阶段里程碑。
  - 失败复盘。
  - 周报/月报。
  - 考前冲刺。
  - 会员到期。
- 支持静默日配置，例如每周日不发送普通提醒。
- 支持考前冲刺频率配置，例如考试前 7 天增加提醒。
- 用户关闭总开关后不创建任何提醒日志。
- 用户只关闭某类提醒时，不影响其他提醒类型。
- 微信未绑定或 Provider 未配置时，记录明确跳过原因，不影响邮件渠道。

#### 一致性与幂等

- 同一用户、目标、渠道、提醒类型、自然日只创建一条有效提醒日志。
- 多个 API 实例同时扫描时不得重复创建。
- Worker 重复消费不能重复发送已经成功的日志。
- 队列失败后根据可重试错误进行指数退避。
- 永久错误直接失败，不占用全部自动重试次数。
- 所有日期判断统一使用用户时区，数据库继续存 UTC 时间。

#### 技术设计要求

- 优先使用 NestJS Schedule 或独立 scheduler process。
- 调度器只负责查找候选并创建日志，不直接调用 Provider。
- 实际发送继续通过 BullMQ `email` 队列和现有 Worker。
- 增加分布式锁或数据库唯一约束，避免多实例重复扫描。
- 增加 `schedulerRunId`、`scheduledFor`、`skipReason` 等可追踪字段，若无需新列则写入结构化 metadata。
- 提供管理员手动触发指定时间窗口扫描的接口，用于补偿和排障。

#### 前端需求

- 账号页增加静默日和考前冲刺提醒设置。
- 展示下一次预计提醒时间。
- 区分自动调度、用户手动预览和管理员补偿生成的日志来源。
- 清晰展示渠道未配置、微信未绑定、达到最大重试次数等状态。

#### 实施清单

- [x] 确定调度器部署形态。
- [x] 设计调度 metadata 或数据库字段。
- [x] 新增静默日和冲刺频率配置。
- [x] 实现时区安全的候选用户扫描。
- [x] 实现多实例幂等和唯一性保护。
- [x] 接入 BullMQ 发送队列。
- [x] 实现管理员补偿触发接口。
- [x] 更新 Web 提醒设置和日志展示。
- [x] 增加单元/集成测试。
- [x] 增加跨时区和重复扫描 E2E。
- [x] 更新 `SPEC.md`、`README.md` 和环境变量示例。

#### Definition of Done

- 自动调度无需用户操作即可创建并发送提醒。
- 同一分钟执行两次扫描不会产生重复提醒。
- 北京时间自然日边界、静默日和关闭开关均有测试。
- Provider 暂时不可用不会丢失提醒任务。
- 管理员能够定位每次扫描和发送失败原因。

### P0-2 会员额度强制执行

**状态：** `DONE`
**目标：** 所有需要消耗额度的入口由后端统一校验，前端展示仅作为用户体验，不作为安全边界。

#### 已确认套餐数值

以下数字已在 ADR-001 中确认，并可通过系统配置或 entitlement 覆盖：

| 能力 | Free 建议值 | Pro 建议值 | 周期 |
| --- | --- | --- | --- |
| 进行中目标 | 1 | 不限 | 即时 |
| 计划生成 | 3 | 30 | 月 |
| 打卡 AI 评分 | 3 | 30 | 日 |
| 评分申诉 | 1 | 10 | 周 |
| 计划重规划 | 1 | 10 | 周 |
| 周/月报告 | 1 | 20 | 月 |
| 奖励卡片 | 5 | 100 | 总量 |
| 上传空间 | 50 MB | 5 GB | 总量 |

#### 功能需求

- 所有额度检查必须在事务内完成，避免并发超额。
- 只有实际创建成功的 Job 或资源才消耗额度。
- 幂等重试、Worker 重试和管理员重试不得重复计费。
- 用户取消未运行 Job 时，明确是否返还额度并保持一致。
- 支付升级后立即生效；到期后新操作按 Free 限制处理，历史数据保留。
- 系统管理员可以调整额度配置，但必须记录审计日志。
- API 超额返回统一错误码、当前用量、上限和重置时间。
- 前端在提交前显示剩余额度，但仍以服务端结果为准。

#### 数据设计

- 评估新增 `entitlements`、`usage_records` 和 `quota_reservations`。
- `usage_records` 至少记录 user、能力代码、数量、关联资源、幂等键、周期和创建时间。
- 对按日、周、月统计的能力建立合适索引。
- 支持系统配置覆盖套餐默认值。

#### 免费版展示策略

- 已采用严格模式：Free 只展示完成状态、基础进度和基础健康状态，不展示 AI 总分、维度、证据分析、总结和建议。
- 后端负责剥离数据，前端隐藏不是安全边界。

### P0-3 真实 AI Provider 补齐

**状态：** `IN_PROGRESS`（代码与自动化验收完成，等待预发布 DeepSeek 密钥执行真实调用）
**目标：** 在保留 Mock 测试能力的同时，为计划之外的关键 AI 链路提供真实 DeepSeek 实现和统一结构化输出验证。

#### Provider 范围

- 目标结构化解析与可行性评估。
- 计划生成与重规划。
- 打卡评分。
- 评分申诉复评。
- 偏差原因总结。
- 救援任务建议与生成。
- 周报/月报文案。
- 失败复盘。

#### 统一 Provider 契约

- 每类 AI 能力定义独立 TypeScript 输入、输出类型和 JSON Schema。
- 输出必须经过结构校验、长度限制和业务范围校验。
- 所有调用记录：
  - `provider`
  - `model`
  - `promptVersion`
  - `inputHash`
  - `requestId`
  - token 用量
  - 延迟
  - 成本估算
  - 重试次数
  - 失败分类
- Prompt 必须版本化，不允许直接在 Service 内散落长字符串。
- 用户输入必须作为数据区而不是系统指令拼接，减少 Prompt Injection 风险。
- 真实 Provider 失败时按能力决定：重试、规则降级、Mock 降级或明确失败。
- 测试环境默认使用 Mock，不调用外部 API。

#### 打卡评分要求

- 使用固定 rubric 计算六个维度。
- 每个维度给出分数和引用用户证据。
- 分数范围、总分计算和证据格式由后端校验。
- 不允许模型直接决定会员可见范围。
- 明显无意义、过短或自相矛盾的输入给出证据不足说明。

#### 申诉要求

- 只允许原任务要求、原打卡和新增事实进入复评上下文。
- 在调用模型前由规则层判断是否存在最低限度新增事实。
- 模型输出必须说明每个变化维度及新增证据。
- 情绪化要求、威胁、讨好和提示注入不得影响评分。
- 无有效新增事实时由规则层直接维持原分，避免无意义调用和成本。

#### 目标解析与可行性评估

- 从自然语言识别目标分类、考试名称、分数、日期、科目和材料。
- 检测时间不足、投入不足、目标差距过大和信息缺失。
- 输出补充问题，每次最多 3 个，优先询问会显著改变计划的信息。
- 对不现实目标进行强提醒，但允许用户确认后继续。
- 所有 AI 自动补全字段必须由用户确认，不得静默覆盖用户输入。

#### 实施清单

- [x] 建立统一 AI 调用和 Schema 校验基础层。
- [x] 增加 AI 调用日志数据模型或扩展现有 Job metadata。
- [x] 将 Prompt 抽离并版本化。
- [x] 实现 DeepSeek Scoring Provider。
- [x] 实现 DeepSeek Appeal Provider。
- [x] 实现 DeepSeek Rescue Provider。
- [x] 实现 DeepSeek Failure Report Provider。
- [x] 实现目标解析和可行性 Provider。
- [x] 接入 Worker、超时和重试策略。
- [x] 增加规则降级与用户错误提示。
- [x] 增加 Schema 非法、超时、限流和空响应测试。
- [x] 使用录制响应完成不依赖真实密钥的契约测试。
- [ ] 在预发布环境完成少量真实调用验证。

#### Definition of Done

- 配置 DeepSeek 后所有目标 AI 能力均能走真实 Provider。
- 无 Key 环境仍能通过完整自动化测试。
- Provider 返回格式异常不会污染数据库。
- 所有调用能在后台追踪版本、耗时、成本和错误。
- 真实 AI 失败时用户能看到明确状态，Worker 不会无限重试。

## 6. P1：安全、存储和可靠性

### P1-1 云对象存储与病毒扫描

**状态：** `IN_PROGRESS`（仓库实现与自动化验收已完成，待生产 S3 私有 Bucket 和 ClamAV 部署联调）

#### 功能需求

- 支持 S3、阿里云 OSS 或腾讯云 COS，具体供应商通过配置选择。
- 保留本地 Provider，仅用于开发和自动化测试。
- 生产环境检测到本地 Provider 时启动失败或输出高优先级告警。
- 上传流程：创建资产 → 获取预签名 URL → 客户端直传 → 完成回调 → 校验 → 扫描 → `READY`。
- 未通过扫描的文件进入 `QUARANTINED` 或 `REJECTED`，不能下载或用于打卡。
- 限制 MIME、扩展名、文件大小、图片尺寸和 PDF 页数。
- 下载使用短时签名 URL，不暴露 Bucket 公共权限。
- 删除目标、资产或账号后异步删除对象并记录结果。
- 提供孤立对象和过期 `PENDING_UPLOAD` 清理任务。

#### 安全要求

- 不信任客户端声明的 MIME 和文件名。
- 使用文件魔数、服务端大小和 SHA-256 复核。
- 对文件名做显示层转义，不作为对象 key。
- 防止 SVG、HTML 等主动内容绕过图片限制。
- 扫描 Provider 超时不得默认放行。
- 管理员查看资产同样需要权限和审计。

#### 实施清单

- [x] 选择 AWS S3 API 作为生产对象存储协议，并允许通过 endpoint 配置兼容 OSS/COS。
- [x] 实现 S3 云存储 Provider；本地 Provider 仅允许开发和测试，生产配置为本地存储时拒绝启动。
- [x] 设计并实现创建、直传、完成回调、服务端复核、异步扫描和就绪状态机。
- [x] 接入 ClamAV `INSTREAM` 扫描协议；超时、协议错误和不可用均不默认放行。
- [x] 增加 BullMQ 扫描/删除/清理 Worker、定时调度、失败重试和孤立对象清理。
- [x] 更新 Web SHA-256 计算、直传进度、完成回调、扫描轮询和失败提示。
- [x] 增加恶意 MIME、魔数不符、EICAR、超限、未扫描引用和跨用户访问测试。
- [x] 增加 S3 SDK 协议往返、正常删除、目标/账号删除任务、过期上传清理和删除失败补偿测试。

#### Definition of Done

- 生产文件不经过 API 进程本地磁盘长期保存。
- 未扫描文件不可被打卡引用或下载。
- 所有对象访问都经过 owner 或管理员权限验证。
- 删除和清理任务可重试、可审计。

## 11. 跨阶段工程规范

### 11.1 API 规范

- 所有写接口必须验证 owner 或 permission。
- 所有列表接口必须有分页或明确上限。
- 错误响应使用稳定错误码，不依赖中文 message 做程序判断。
- 时间字段使用 ISO 8601，数据库保存 UTC，业务自然日明确时区。
- 幂等接口必须定义幂等键和重复调用结果。
- 外部 Webhook 保留原始事件 ID 和处理状态。

### 11.2 数据库规范

- 每次 Schema 修改必须附 Prisma migration。
- 迁移必须考虑已有数据，不允许假设空数据库。
- 大表新增非空字段使用分阶段迁移。
- 唯一约束、外键和索引需要和业务幂等规则一致。
- 删除策略明确区分硬删除、软删除和审计保留。

### 11.3 Worker 规范

- 数据库是事实来源，队列不是唯一状态存储。
- Worker 必须幂等，终态任务重复消费直接跳过。
- 重试需要区分临时错误和永久错误。
- 每类任务定义最大次数、backoff 和终止状态。
- 管理员重试必须记录原因和审计日志。
- 用户看到的状态必须来自持久化 Job，而不是仅查询 BullMQ。

### 11.4 前端规范

- 所有异步操作显示 loading、成功、失败和可恢复状态。
- 权限和会员 gating 由统一组件或 hook 处理。
- 页面刷新后从后端恢复状态，不依赖临时内存数据。
- 支持键盘焦点、`prefers-reduced-motion` 和清晰错误提示。
- 390px 移动视口不得水平溢出。
- 普通用户永远不展示后台入口。

### 11.5 测试规范

### ADR-001：采用服务端原子配额与严格 Free AI 隐藏

- 日期：2026-06-19
- 状态：ACCEPTED
- 背景：旧实现只强制进行中目标限制，其他 AI 数字仅统计展示，且 Free 仍返回基础总分，与完整版规格冲突。
- 决策：采用本文 P0-2 表格中的建议额度；Free 完全隐藏 AI 总分和详细分析；用 `quota_buckets` 原子扣减、`usage_records` 追踪、`entitlements` 覆盖；UTC 作为当前计费周期边界。
- 备选方案：继续按历史表实时 count；仅在前端禁用；Free 保留基础总分。以上方案无法可靠处理并发、绕过前端和规格一致性，因此不采用。
- 影响：部分原本可以高频调用 Mock AI 的 Free 用户会收到 HTTP 429；已有用量由 migration 回填；Web 展示更多真实额度。
- 后续动作：P2-3 将支付订阅写入 entitlement；若需要用户时区计费，新增 ADR 和兼容迁移。

## 16. 建议执行顺序

在没有新的业务优先级变化时，严格按以下顺序实施：

1. P0-1 自动提醒调度。
2. P0-2 会员额度强制执行。
3. P0-3 真实 AI Provider 补齐。
4. P1-3 可观测性、备份与故障恢复基础。
5. P1-1 云对象存储与病毒扫描。
6. P1-2 敏感数据加密与 AI 脱敏。
7. P2-3 订阅与会员权益模型。
8. P2-1 Stripe 正式支付。
9. P2-2 微信支付正式接入。
10. P3-1 AI 目标创建助手。
11. P3-2 统一成长事件时间线。
12. P3-3 后台三级权限。
13. P4-1 微信登录和账号绑定。
14. P4-2 微信小程序页面。
15. P5-1 性能和容量验证。
16. P5-2 发布和灰度运营验收。

选择该顺序的原因：先让已有 Web 闭环能够自动运行并正确计费，再解决生产安全和可靠性，然后接入正式商业化，最后扩展小程序和进行发布验收。这样可以避免在核心 API、权限和数据模型仍频繁变化时过早维护第二套客户端。
